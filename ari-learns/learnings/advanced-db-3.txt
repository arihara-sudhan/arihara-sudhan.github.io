1. Object and Object-Relational Databases
𖦹 Object-oriented data models have features now incorporated in relational databases and SQL standards.
𖦹 Some features are also in NoSQL and XML models.
𖦹 Object databases (ODBs) were originally called object-oriented databases (OODBs).
𖦹 Traditional models like network, hierarchical, and relational work well for business applications but have limitations for complex applications (CAD/CAM, biology, GIS, multimedia).
𖦹 ODBs allow specifying complex object structures and operations.
𖦹 Growth of object-oriented programming languages (C++, Java) increased the need for ODBs.
𖦹 Object-relational databases (ORDBMS) combine relational systems with object database features.
𖦹 SQL:2008 includes many object features originally proposed for SQL/Object.
𖦹 Many experimental and commercial ODB systems were created but did not gain widespread use due to relational and object-relational popularity.
𖦹 ODMG 3.0 standard was proposed to provide a standard model and language for ODBs.
𖦹 Key ODB concepts include object identity, structure, encapsulation, methods, persistence, and inheritance.
𖦹 SQL standards incorporated these concepts, with object features in SQL:1999 and SQL:2008.

2. Introduction to Object-Oriented Concepts
𖦹 OO concepts originated in OO programming languages (OOPLs) like SIMULA and Smalltalk.
𖦹 Pure OO languages explicitly designed for OO, e.g., Smalltalk.
𖦹 Hybrid OO languages integrate OO features into existing languages, e.g., C++.
𖦹 Objects have state (value) and behavior (operations).
𖦹 In OOPLs, objects are transient; OO databases store objects persistently.
𖦹 Persistent objects can be shared among multiple programs.
𖦹 Internal structure includes instance variables (attributes) and operations (methods).
𖦹 Complete encapsulation can be relaxed to allow attribute access for queries.
𖦹 Operations have a signature (interface) and a method (implementation).
𖦹 Type and class hierarchies allow inheritance, reusing definitions.
𖦹 Relationships among objects can be explicitly represented with inverse references.
𖦹 Operator overloading (polymorphism) allows operations on different object types, often requiring late binding.

3. Object Identity and Literals
𖦹 Each object in an ODB has a unique object identifier (OID).
• OID is immutable and system-generated.
• OID must not depend on attribute values.
• Early ODBs used physical addresses as OIDs.
𖦹 ODBs distinguish between objects (with OID) and literals (without OID).
𖦹 Literals cannot be referenced from other objects but can be part of objects.

4. Complex Type Structures
𖦹 Objects and literals can have complex types.
𖦹 Three main type constructors:
• Atom: basic types (integer, string, boolean) - atomic values.
• Struct/Tuple: structured types made of multiple components.
• Collection: multivalued types (set, list, bag, array, dictionary) containing elements of the same type.
𖦹 Struct constructor generates multiple structured types.
𖦹 Collection types allow ordered or unordered groups of elements.
𖦹 Dictionary constructor creates key-value pairs.
𖦹 Type constructors define data structures for OO database schemas.

5. Encapsulation of Operations and Persistence
𖦹 Encapsulation defines the interface and hides implementation of object operations.
𖦹 Operations can create, destroy, update, or retrieve object state.
𖦹 Interface (signature) is visible; method (implementation) is hidden.
𖦹 Objects can have visible and hidden attributes.
𖦹 Operations are applied using dot notation (e.g., d.no_of_emps).
𖦹 Persistence mechanisms:
• Naming: assign a unique persistent name to an object.
• Reachability: objects reachable from a persistent object also become persistent.
𖦹 Extent is a named collection of objects of a class that are persistent.
𖦹 Multiple persistent collections can exist for the same class.

6. Type Hierarchies and Inheritance
𖦹 ODBs support subtypes and supertypes forming type hierarchies.
𖦹 Subtypes inherit attributes and operations from supertypes.
• Example: PERSON type has Name, Address, Birth_date, Age, Ssn.
• EMPLOYEE subtype adds Salary, Hire_date, Seniority.
• STUDENT subtype adds Major, Gpa.
𖦹 Inheritance allows incremental development and reusability of types.
𖦹 Area operation for GEOMETRY_OBJECT can have different implementations for RECTANGLE, TRIANGLE, and CIRCLE.
𖦹 Objects belong to declared types and persistent collections (extents).
𖦹 Constraint: objects in a subtype extent must also belong to supertype extent.
𖦹 ROOT or OBJECT class can include all objects in the system.

7. Object Database Extensions to SQL
𖦹 SQL was first specified by Chamberlin and Boyce (1974) and standardized in 1989 and 1992.
𖦹 SQL3 (later SQL:99) incorporated object database features; initially called SQL/Object, later part of SQL/Foundation in SQL:2008.
𖦹 Object-relational model combines relational model with object database enhancements.
𖦹 Revisions in 2003 and 2006 added XML-related features.
𖦹 Object database features include:
• Type constructors for complex objects: row type, array type, set, list, bag (added in SQL:2008)
• Object identity via reference type
• Encapsulation via user-defined types (UDTs) with operations; user-defined routines (UDRs)
• Inheritance using <pink>UNDER</pink> keyword

9. Object Identifiers Using Reference Types
𖦹 Unique object identifiers created using <pink>REF</pink>.
𖦹 Syntax: <pink>REF IS SYSTEM GENERATED</pink> creates system-generated ID.
𖦹 User can specify <pink>REF IS <OID_ATTRIBUTE> <VALUE_GENERATION_METHOD></pink> with SYSTEM GENERATED or DERIVED.

10. Creating Tables Based on the UDTs
𖦹 INSTANTIABLE UDTs can be used to create tables.
𖦹 Noninstantiable UDTs only serve as attribute types.
𖦹 Attributes like PERSON_ID hold system-generated object IDs for table rows.

11. Encapsulation of Operations
𖦹 UDTs can have methods along with attributes.
𖦹 Syntax: <pink>CREATE TYPE</pink> TYPE_NAME (<attributes>, <methods>)
𖦹 Constructor TYPE_T() returns new object with default values.
𖦹 Observer function reads attribute; mutator updates it.
𖦹 Functions may require <pink>EXECUTE</pink> privilege.
𖦹 Functions can be internal (SQL PSM) or external (host language signature only).
𖦹 Attributes/functions categories: PUBLIC, PRIVATE, PROTECTED.
𖦹 Virtual attributes computed via functions.

12. Specifying Inheritance and Overloading of Functions
𖦹 Inheritance applied using <pink>UNDER</pink> for types or tables.
𖦹 <pink>NOT FINAL</pink> allows creation of subtypes.
𖦹 Rules:
• All attributes inherited
• Supertype order sets hierarchy
• Subtype instance usable where supertype instance is used
• Subtype can redefine functions with same signature
• Function calls resolve best match based on argument types
• Dynamic linking considers parameter types at runtime
𖦹 Table inheritance: INSERT, DELETE, UPDATE in subtable propagate to supertable; tuples in subtable must exist in supertable.

13. Specifying Relationships via Reference
𖦹 Component attribute can reference tuple of another table using <pink>REF</pink>.
𖦹 <pink>SCOPE</pink> specifies table whose tuples can be referenced.
𖦹 Dot notation accesses components; <pink>–></pink> used for dereferencing REF attributes.
𖦹 Example:
<pink>SELECT</pink> E.Employee–>NAME
<pink>FROM</pink> EMPLOYMENT AS E
<pink>WHERE</pink> E.Company–>COMP_NAME = ‘ABCXYZ’;
𖦹 <pink>–></pink> works like in C: r–>a returns attribute a of referenced tuple.
𖦹 <pink>SCOPE</pink> allows reference to tuples within a specific table of same type.

14. ODMG Object Model and Object Definition Language ODL
𖦹 ODMG-93 (1.0) standard created by ODMG consortium; revised to 2.0 and 3.0.
𖦹 Standard includes object model, object definition language (ODL), object query language (OQL), and bindings to OOP languages.
𖦹 ODMG 3.0 based on decades of research in conceptual modeling and object databases.
𖦹 Object-relational technology incorporates object concepts into SQL.

15. Overview of ODMG Object Model
𖦹 Provides standard data model for object databases, similar to SQL for relational DBs.
𖦹 Establishes standard terminology in the field.
𖦹 Basic building blocks: objects and literals.
• Objects have identifier and state; literals have state only.
𖦹 Object aspects: identifier, name, lifetime, structure, creation.
- Identifier: unique Object_id.
- Name: optional, unique within ODB; used to locate object.
- Lifetime: persistent or transient; independent of class/type.
- Structure: atomic (single user-defined object) or non-atomic (collection of objects).
- Creation: via <pink>new</pink> operation in Object_Factory interface.

16. Types of Literals
𖦹 Atomic literals: predefined basic data types (long, short, unsigned, float, double, boolean, char, string, enum).
𖦹 Structured literals: user-defined using <pink>STRUCT</pink>, include Date, Interval, Time, Timestamp.
𖦹 Collection literals: collections of objects or values without Object_id; type generators include set<T>, bag<T>, list<T>, array<T>, dictionary<K,V>.

17. Object Interfaces and Classes
𖦹 Interface: defines behavior only, non-instantiable.
𖦹 Class: defines both state and behavior, instantiable.
𖦹 Literal declaration: defines state only, no behavior or Object_id.
𖦹 All objects inherit Object interface operations: <pink>copy()</pink>, <pink>delete()</pink>, <pink>same_as()</pink>.
- Dot notation example: O.<pink>same_as</pink>(P), P = O.<pink>copy</pink>().
- Arrow notation alternative: O–><pink>same_as</pink>(P), O–><pink>copy</pink>().

18. Inheritance in ODMG
𖦹 Behavior-only inheritance: interface inheritance (ISA), uses colon (:), supertype must be interface, subtype can be class or interface.
𖦹 EXTENDS inheritance: inherits state + behavior among classes, uses <pink>extends</pink>, only single inheritance allowed.
𖦹 Classes can inherit behavior from multiple interfaces via colon.

19. Built-in Interfaces and Classes
𖦹 All inherit Object interface.
𖦹 Collection objects include set, list, bag, array, dictionary; provide operations like <pink>cardinality()</pink>, <pink>is_empty()</pink>, <pink>insert_element(E)</pink>, <pink>remove_element(E)</pink>, <pink>contains_element(E)</pink>, <pink>create_iterator()</pink>.
𖦹 Iterator interface: <pink>reset()</pink>, <pink>next_position()</pink>, <pink>get_element()</pink>.
𖦹 Exceptions: ElementNotFound, NoMoreElements.
𖦹 Specialized operations for set<T>: <pink>create_union(S)</pink>, <pink>is_subset_of(S)</pink>.
𖦹 List<T> operations: <pink>insert_element_first(E)</pink>, <pink>insert_element_after(E,I)</pink>, <pink>remove_first_element()</pink>, <pink>concat(I)</pink>, <pink>append(I)</pink>.
𖦹 Array<T> operations: <pink>replace_element_at(I,E)</pink>, <pink>remove_element_at(I)</pink>, <pink>retrieve_element_at(I)</pink>, <pink>resize(N)</pink>.
𖦹 Dictionary<K,V> operations: <pink>bind(K,V)</pink>, <pink>unbind(K)</pink>, <pink>lookup(K)</pink>, <pink>contains_key(K)</pink>.

20. Atomic (User-Defined) Objects
𖦹 Defined using <pink>class</pink> in ODL; not collections.
𖦹 Example: STUDENT, EMPLOYEE, DEPARTMENT.
𖦹 Components:
• Attributes: simple or complex literals, or Object_ids.
• Relationships: binary, use <pink>relationship</pink> keyword with <pink>inverse</pink> for bidirectional mapping.
• Operations: define behavior, can include exceptions.

21. Extents, Keys, and Factory Objects
𖦹 Extent: declared with <pink>extent</pink>, contains all persistent objects of a class.
𖦹 Keys: unique attributes or relationships per extent; composite keys in parentheses.
𖦹 Factory objects: create individual objects using <pink>new()</pink>.
• Examples: ObjectFactory, SetFactory, ListFactory, ArrayFactory, DictionaryFactory, DateFactory.
𖦹 Database interfaces: DatabaseFactory and Database.
- Operations: <pink>open(name)</pink>, <pink>close()</pink>, <pink>bind(object,name)</pink>, <pink>unbind(name)</pink>, <pink>lookup(name)</pink>.

22. Object Definition Language (ODL)
𖦹 ODL used to create object specifications (classes, interfaces) for object database schema.
𖦹 Independent of programming language; can be mapped via language bindings (C++, Java, Smalltalk).
𖦹 Example: UNIVERSITY database schema.
• Entity types mapped to ODL classes.
• Inheritance via <pink>extends</pink>; no direct mapping for union types or multiple inheritance.
• Classes: PERSON, FACULTY, STUDENT, GRAD_STUDENT.
• Extents: PERSONS, FACULTY, STUDENTS, GRAD_STUDENTS.
• Subsets: STUDENTS ⊆ PERSONS, GRAD_STUDENTS ⊆ STUDENTS.
• Individual objects inherit properties and operations of supertypes.

38. Differences between ODB and RDB Design
𖦹 ODB handles relationships using relationship properties or reference attributes containing OID(s) of related objects.
• Single references and collections of references are allowed.
• References can be declared in one or both directions; inverses enforce referential integrity.
𖦹 RDB handles relationships using attributes with matching values (foreign keys).
• Foreign keys are single-valued.
• M:N relationships require separate tables.
𖦹 Mapping binary relationships with attributes in ODB is complex.
• Including attributes in both directions causes redundancy and possible inconsistencies.
• Separate classes may be used for relationships, especially for n-ary relationships (n > 2).
𖦹 Inheritance is built into ODB using constructs like derived (:) and extends.
• RDB has no built-in inheritance; options must be chosen manually.
𖦹 Operations must be specified early in ODB design as part of class definitions.
• RDB can delay operation specifications until implementation.
𖦹 Philosophical difference: ODB requires predefined behaviors, while RDB allows ad hoc queries.

39. Mapping EER Schema to ODB Schema Overview
𖦹 Type declarations of object classes are straightforward for EER schemas without categories or n-ary relationships (n > 2).
𖦹 Class operations are not in the EER diagram and must be added after structural mapping.

40. Step 1: Create ODL Classes
𖦹 Create an ODL class for each EER entity type or subclass.
• Include all attributes from the EER class.
• Multivalued attributes use set, bag, or list constructors.
- list: ordered values, bag: allows duplicates, set: unique values.
• Composite attributes use struct (tuple) constructor.
• Declare extent and specify key attributes.

41. Step 2: Add Relationship Properties
𖦹 Add relationship properties or reference attributes for each binary relationship.
• Can be created in one or both directions.
• Inverse references can be declared if supported.
𖦹 Single-valued for 1:1 or N:1 relationships; collection types for 1:N or M:N.
𖦹 Relationship attributes can use struct<reference, attributes>.
- Using both directions may cause redundancy.
- Some ODBMS may not support inverse relationships; programmers must maintain relationships in code.

42. Step 3: Include Operations
𖦹 Add operations for each class based on requirements.
• Constructor checks constraints when creating objects.
• Destructor checks constraints when deleting objects.
• Other methods include additional constraint checks.
- Program code should enforce these checks.

43. Step 4: Subclass Mapping
𖦹 ODL class corresponding to a subclass inherits attributes, relationships, and methods from its superclass using extends.
𖦹 Local attributes, relationships, and operations are added as in steps 1, 2, and 3.

44. Step 5: Weak Entity Types
𖦹 Map weak entity types like regular entities.
• Alternative: map weak entities with only identifying relationship as composite multivalued attributes using set<struct<…>> or list<struct<…>>.
• Attributes included in struct correspond to tuple constructor.

45. Step 6: Categories (Union Types)
𖦹 Mapping categories is difficult.
• Create a class for the category.
• Define 1:1 relationships between the category and each superclass, similar to EER-to-relational mapping.

46. Step 7: N-ary Relationships and M:N
𖦹 N-ary relationships (n > 2) are mapped into a separate class.
• Include references to each participating class using 1:N mapping.
𖦹 M:N binary relationships with attributes may also be mapped as separate classes if desired.

𖦹 The mapping algorithm is applied to a subset of the UNIVERSITY database.
𖦹 The resulting ODL object schema follows the ODMG standard.

48. The Object Query Language OQL Overview
𖦹 OQL is the query language for the ODMG object model.
• Works with programming languages like C++, Smalltalk, and Java via ODMG bindings.
• Queries return objects matching the language's type system.
• Class operations can have code in these programming languages.
𖦹 OQL syntax is similar to SQL but includes features for object identity, complex objects, operations, inheritance, polymorphism, and relationships.
𖦹 Examples use the UNIVERSITY database schema.

49. Simple OQL Queries, Entry Points, and Iterator Variables
𖦹 Basic OQL query syntax: select … from … where …
• Example:

<pink>from</pink> D <pink>in</pink> DEPARTMENTS
<pink>where</pink> D.College = 'Engineering';
𖦹 Database entry points can be any named persistent object or extent.
• Extent names refer to persistent collections (set of objects), e.g., DEPARTMENTS: set<DEPARTMENT>.
𖦹 Iterator variables range over each object in the collection.
• Example: D in DEPARTMENTS.
𖦹 Result type: bag for select, set for select distinct (removes duplicates).
𖦹 Iterator variable syntax options:
• D in DEPARTMENTS
• DEPARTMENTS D
• DEPARTMENTS AS D

50. Query Results and Path Expressions
𖦹 Query results can be any ODMG object model type.

58. Overview of the C++ Language Binding in ODMG
𖦹 C++ language binding maps ODL constructs to C++ constructs using a class library.
• Object Manipulation Language (OML) specifies how database objects are retrieved and manipulated in C++.
• Physical pragmas provide control over storage, clustering, indexes, and memory.

• D_Set<T>, D_List<T>, D_Bag<T>, D_Varray<T>, D_Dictionary<T>
- Example: D_Set<D_Ref<STUDENT>> is a set of references to STUDENT objects; D_Set<string> is a set of strings.

𖦹 OML overloads <pink>new</pink> to create persistent or transient objects.

64. Techniques used internally to process high-level queries
𖦹 A high-level query must be scanned, parsed, and validated.
𖦹 The scanner identifies tokens such as keywords, attribute names, and relation names.
𖦹 The parser checks if the syntax follows the query language rules.
𖦹 Validation ensures all attribute and relation names exist and match the database schema.
𖦹 An internal representation is created, usually as a query tree or query graph.
• Query tree is a tree structure.
• Query graph is a directed acyclic graph (DAG).
𖦹 The DBMS creates an execution strategy or query plan for retrieving results.
𖦹 Choosing a suitable execution strategy is called query optimization.

65. Execution steps for processing a query
𖦹 The query optimizer produces a good execution plan.
𖦹 The code generator creates the code to run the plan.
𖦹 The runtime processor executes the query code in compiled or interpreted form.
𖦹 Runtime errors generate error messages.
𖦹 Code can be executed immediately or stored for later use.

66. Nature of query optimization
𖦹 Optimization does not always find the absolute best plan.
𖦹 Optimal plans can be too time-consuming to compute.
𖦹 Accurate information about table sizes and value distributions may be missing.
𖦹 Estimating expected result sizes is required based on query predicates.
𖦹 Planning a good strategy is more realistic than finding a perfect one.

67. Role of navigational and high-level languages
𖦹 In navigational languages, the programmer chooses the execution strategy.
𖦹 DBMS has limited optimization capability for navigational languages.
𖦹 High-level languages like SQL or OQL specify what result is needed, not how to obtain it.
𖦹 Optimization is necessary for high-level declarative queries.

68. Query processing and optimization in RDBMS
𖦹 RDBMS evaluates alternative execution strategies systematically.
𖦹 The system uses various access algorithms to implement SELECT, JOIN, and other operations.
𖦹 Only strategies supported by available algorithms and the physical design are considered.
𖦹 SQL queries are translated into relational algebra and additional operations before optimization.

70. Techniques of query optimization
𖦹 Two main techniques are used:
• Heuristic rules to reorder operations in a query tree.
• Cost estimation to choose the plan with minimum estimated cost.
𖦹 These require prior understanding of SQL, relational algebra, file structures, and indexing.

71. Translating SQL queries into relational algebra
𖦹 SQL queries are translated into equivalent extended relational algebra expressions.
𖦹 The expression is represented as a query tree that is optimized later.
𖦹 SQL queries are divided into query blocks for translation and optimization.
𖦹 A query block contains SELECT-FROM-WHERE and may include GROUP BY and HAVING.
𖦹 Nested queries are separated into multiple query blocks.
𖦹 Aggregate operators like MAX, MIN, SUM, COUNT are supported in extended algebra.

72. Example of nested query translation
𖦹 SQL query retrieves employees whose salary is greater than the maximum salary in department 5.
𖦹 Query is decomposed into inner and outer blocks.
• Inner block retrieves MAX(Salary) where Dno = 5.
• Outer block selects Lname, Fname where Salary > result of inner block.
𖦹 Inner block translation:
ℑMAX Salary(σDno=5(EMPLOYEE))
𖦹 Outer block translation:
πLname,Fname(σSalary>c(EMPLOYEE))
- c is the result from the inner block.
𖦹 The inner block is evaluated once because it is uncorrelated.
𖦹 Correlated subqueries are more complex and need special optimization techniques.

73. Semi-join and anti-join operators
𖦹 SQL queries for enterprise and OLAP systems may use operations beyond standard relational algebra.
𖦹 Semi-join and anti-join are two common additional operators.
𖦹 Semi-join is used for unnesting EXISTS, IN, and ANY subqueries.
- Notation: T1.X S = T2.Y
- Semantics: Return T1 row when a match is found with any T2 value; no need to find all matches.

74. Semi-join example
𖦹 Schema:
EMPLOYEE(Ssn, Bdate, Address, Sex, Salary, Dno)
DEPARTMENT(Dnumber, Dname, Dmgrssn, Zipcode)
𖦹 Query counts departments with employees earning more than 200000.
𖦹 Original nested subquery retrieves E.Dno values where Salary > 200000.
𖦹 After unnesting, semi-join is used:
SELECT COUNT(*)
FROM EMPLOYEE E, DEPARTMENT D
WHERE D.Dnumber S= E.Dno AND E.Salary > 200000;
- Semi-join finds department numbers that match employee Dno values.

75. Anti-join operator
𖦹 Anti-join is used for unnesting NOT EXISTS, NOT IN, and ALL subqueries.
- Notation: T1.X A = T2.Y
- Semantics: Reject T1 row if a match exists in T2; return only if no match is found.

76. Anti-join example
𖦹 Query counts employees not working in departments with Zipcode = 30332.
𖦹 Anti-join required because inner join would give incorrect results.
𖦹 Unnested form using anti-join:
SELECT COUNT(*)
FROM EMPLOYEE, DEPARTMENT
WHERE EMPLOYEE.Dno A= DEPARTMENT AND Zipcode =30332
- Anti-join returns employee rows whose Dno does not match any department with Zipcode 30332.

77. Algorithms for External Sorting
𖦹 External sorting is used when files are too large to fit in main memory.
𖦹 Used for ORDER BY, sort-merge JOIN, UNION, INTERSECTION, and DISTINCT.
𖦹 Sorting may be skipped if an index already provides ordered access.

78. Sort-Merge Strategy Overview
𖦹 Large file is broken into smaller sorted runs.
𖦹 Runs are created by loading pieces into memory, sorting them, and writing back.
𖦹 Sorted runs are then merged into larger sorted subfiles until one sorted file remains.

79. Buffer Usage in External Sorting
𖦹 Main memory buffer space is part of the DBMS cache.
𖦹 Memory is divided into buffers equal in size to one disk block.
𖦹 Each buffer holds exactly one disk block.

80. Sorting Phase
𖦹 Load as many blocks as fit into nB buffers.
𖦹 Sort the records in memory.
𖦹 Write as temporary sorted subfile (run).
• Number of runs nR = ceil(b / nB).
• Example: nB = 5, b = 1024 → nR = 205 runs.

81. Merging Phase
𖦹 Runs are merged in multiple passes.
𖦹 In each merge step, one buffer per run + one output buffer is needed.
𖦹 Degree of merge dM = min(nB − 1, nR).
𖦹 Number of merge passes = ceil(log_dM(nR)).
• Example: nB = 5 → dM = 4 → multiple passes until one sorted file.

82. Example Merge Progress (nB = 5, nR = 205)
𖦹 First pass: 205 runs → merge 4 at a time → 52 runs.
𖦹 Second pass: 52 → 13 runs.
𖦹 Third pass: 13 → 4 runs.
𖦹 Final pass: 4 → 1 fully sorted file.

83. Pseudocode for Sort-Merge Algorithm
𖦹 Sorting phase:
• Read next k blocks (k = nB) or remaining blocks.
• Sort in memory.
• Write sorted subfile.
𖦹 Merging phase:
• Read k−1 subfiles block by block.
• Merge and write output subfile.
• Repeat until one subfile remains.

<pink>set</pink> <yellow>i ← 1</yellow>;
<pink>set</pink> <yellow>j ← b</yellow>;
<pink>set</pink> <yellow>k ← nB</yellow>;
<pink>set</pink> <yellow>m ← ceil(j / k)</yellow>;
<pink>while</pink> <yellow>(i ≤ m)</yellow> <pink>do</pink> {
  <yellow>read next k blocks</yellow>;
  <yellow>sort and write subfile</yellow>;
  <yellow>i ← i + 1</yellow>;
}
<pink>set</pink> <yellow>i ← 1</yellow>;
<pink>set</pink> <yellow>p ← ceil(log(k−1)(m))</yellow>;
<pink>set</pink> <yellow>j ← m</yellow>;
<pink>while</pink> <yellow>(i ≤ p)</yellow> <pink>do</pink> {
  <yellow>n ← 1</yellow>;
  <yellow>q ← ceil(j / (k−1))</yellow>;
  <pink>while</pink> <yellow>(n ≤ q)</yellow> <pink>do</pink> {
    <yellow>read k−1 subfiles</yellow>;
    <yellow>merge and write</yellow>;
    <yellow>n ← n + 1</yellow>;
  }
  <yellow>j ← q</yellow>;
  <yellow>i ← i + 1</yellow>;
}

84. Cost of Sort-Merge Algorithm
𖦹 Approximate cost = (2 * b) + (2 * b * (log_dM nR)).
𖦹 First term: reading + writing during sorting phase.
𖦹 Second term: reading + writing during each merge pass.

85. Worst-Case Performance
𖦹 Minimum buffers required: nB = 3.
𖦹 Then dM = 2.
𖦹 Cost becomes: (2 * b) + (2 * b * log₂(nR)).

86. Algorithms for SELECT operation
𖦹 SELECT is a search to find records that satisfy a condition.
𖦹 Some algorithms require specific access paths.
𖦹 Example operations: OP1 to OP7.
• OP1: σSsn = ‘123456789’ (EMPLOYEE)
• OP2: σDnumber > 5 (DEPARTMENT)
• OP3: σDno = 5 (EMPLOYEE)
• OP4: σDno = 5 AND Salary > 30000 AND Sex = ‘F’ (EMPLOYEE)
• OP5: σEssn = ‘123456789’ AND Pno = 10 (WORKS_ON)
• OP6: SELECT * FROM EMPLOYEE WHERE Dno IN (3,27,49)
• OP7: SELECT First_name, Lname FROM Employee WHERE ((Salary*Commission_pct)+Salary)>15000

87. Search methods for simple selection (file scans)
𖦹 S1: Linear search. Scan all blocks, test each record.
𖦹 S2: Binary search. Used only if file is ordered on key. Example: OP1 if ordered on Ssn.
𖦹 S3a: Use primary index for equality on key. Example: Ssn='123456789'.
𖦹 S3b: Use hash key for equality on key. Example: Ssn='123456789'.
𖦹 S4: Primary index for ranges. Example: Dnumber > 5 retrieves records after Dnumber=5.
𖦹 S5: Clustering index for equality on nonkey. Example: Dno=5 retrieves all matching records.
𖦹 S6: Secondary B+-tree index. Supports equality and range queries.
• Range example: 3000 <= Salary <= 4000.
𖦹 S7a: Bitmap index. OR bitmaps for values. Example: Dnumber IN (3,27,49) in OP6.
𖦹 S7b: Functional index. Supports expression-based selection.
• Example functional index:
<pink>CREATE</pink> <pink>INDEX</pink> <yellow>income_ix</yellow>
<pink>ON</pink> <yellow>EMPLOYEE (Salary + (Salary*Commission_pct))</yellow>

88. Notes on index-based methods
𖦹 S1 works always.
𖦹 S2 needs sorted file.
𖦹 S3–S6 need indexes.
𖦹 S4 and S6 handle range queries.
𖦹 S7a for enumerated sets.
𖦹 S7b for functional attribute matches.

89. Search methods for conjunctive selection (AND conditions)
𖦹 For OP4, DBMS may use extra methods.
𖦹 S8: Use index on one attribute, then test other conditions.
𖦹 S9: Use composite index if equality conditions match composite key.
• Example: Index on (Essn, Pno) for OP5.
𖦹 S10: Intersection of record pointers from multiple secondary indexes.
• Each index returns pointers; intersect them to find qualifying records.
• Bitmap and functional indexes can also be AND-ed.

90. Choosing access paths for conjunctive selection
𖦹 If only one condition has an access path, use it.
𖦹 Otherwise, S1 is fallback.
𖦹 Optimizer tries to retrieve fewest records at lowest cost.

91. Search methods for disjunctive selection (OR conditions)
𖦹 Harder to optimize than AND conditions.
𖦹 Example: OP4′ = (Dno=5 OR Salary>30000 OR Sex='F').
𖦹 Records satisfying OR are union of results.
𖦹 If any condition lacks an access path → must use linear scan.
𖦹 If all conditions have access paths → combine their record id sets using union.

92. Estimating selectivity
𖦹 Optimizer uses catalog statistics to estimate cost.
𖦹 Catalog stores:
• Number of records |r(R)|
• Tuple width
• Number of blocks bR
• Blocking factor bfr
• Distinct values NDV(A,R)
• min/max values for attributes
𖦹 Selectivity sl = qualifying_records / total_records.

93. Selectivity estimation examples
𖦹 Equality on key: sl = 1 / |r(R)|.
𖦹 Equality on nonkey with i distinct values: sl = 1/i.
𖦹 Range A ≥ v:
• sl = 0 if v > max(A,R)
• sl = (max(A,R) − v) / (max(A,R) − min(A,R))
𖦹 Estimated qualifying records = |r(R)| * sl.

94. Non-uniform distribution issues
𖦹 NDV-based estimation assumes uniform distribution.
𖦹 Real data often non-uniform.
𖦹 Histograms in catalog provide more accurate estimates.

95. Implementing the JOIN operation
𖦹 JOIN is one of the most expensive operations in query processing.
𖦹 Focus is on EQUIJOIN and NATURAL JOIN; join here means one of these.
𖦹 Two-way join is between two files; multiway joins become complex due to many possible join orders.
𖦹 General join form: R A=B S
• A and B must be domain-compatible attributes of R and S.
𖦹 Techniques discussed apply mainly to two-way joins but extend to general join forms.
𖦹 Example operations:
• OP6: EMPLOYEE Dno=Dnumber DEPARTMENT
• OP7: DEPARTMENT Mgr_ssn=Ssn EMPLOYEE

96. Methods for Implementing Joins
𖦹 J1—Nested-loop join (or nested-block join)
• Default brute-force method; no special access paths required.
- For each record t in R (outer loop), check every record s in S (inner loop) for t[A] = s[B].

𖦹 J2—Index-based nested-loop join
• Uses index or hash key on one join attribute (e.g., B of S).
- For each record t in R, retrieve matching records s from S using index: s[B] = t[A].

𖦹 J3—Sort-merge join
• Requires R and S to be sorted by join attributes A and B.
- Scan both files concurrently to match records with equal A and B values.
• If unsorted, use external sorting first.
• Variation: use secondary indexes to access records in join order.
- Inefficiency: physical scattering of records may require multiple disk accesses.

𖦹 J4—Partition-hash join (hash-join)
• Partition R and S using same hash function h on A (R) and B (S).
• Partitioning phase: smaller file R is hashed into memory buckets.
• Probing phase: hash S records to appropriate bucket and combine with R.
- Simplified assumption: smaller file fits entirely in memory; general case removes this restriction.
• Implementation reads disk blocks, not individual records, adjusting to memory buffers.

𖦹 Sort-merge join algorithm example
• Sort R by A and S by B.
• Set i ←1, j ← 1; while i ≤ n and j ≤ m
• Compare R(i)[A] and S(j)[B]:
- If R(i)[A] > S(j)[B], j ← j + 1
- If R(i)[A] < S(j)[B], i ← i + 1
- If equal:
• Output <R(i), S(j)> to T
• Check for additional matching tuples in R and S
• Increment i, j accordingly

𖦹 Projection with duplicate elimination
• Create T′ with projected attributes from R.
- If attribute list includes key of R, T ← T′
- Else: sort T′ and eliminate duplicates using indices i, j.

𖦹 Set operations with sort-merge
• UNION: sort R, S; compare tuples; output unique tuples to T.
• INTERSECTION: sort R, S; output tuples present in both.
• SET DIFFERENCE: sort R, S; output tuples in R not in S.

97. Buffer Space and Outer-Loop Choice in Nested-Loop Join
𖦹 Available buffer space affects join performance, especially in nested-loop join (J1).
• Example: nB = 7 buffers, DEPARTMENT: rD = 50 records, bD = 10 blocks, EMPLOYEE: rE = 6,000 records, bE = 2,000 blocks.
- Outer loop should use the file with fewer blocks for efficiency.
• Inner loop reads one block at a time; matched records are stored in result buffer.
- Result buffer is written to disk when full and reused.

𖦹 Disk block calculations
• EMPLOYEE as outer loop: total block accesses = bE + (⎡bE/(nB-2)⎤ * bD) = 2000 + (⎡2000/5⎤ * 10) = 6000
• DEPARTMENT as outer loop: total block accesses = bD + (⎡bD/(nB-2)⎤ * bE) = 10 + (⎡10/5⎤ * 2000) = 4010
- Writing result file: add bRES block accesses.

98. Join Selection Factor and Performance
𖦹 Join selection factor affects performance of index-based nested-loop join (J2).
• Example OP7: DEPARTMENT joined with EMPLOYEE for manager.
• Two options:
- Retrieve each EMPLOYEE, use index on DEPARTMENT: bE + (rE * (xMgr_ssn + 1)) = 2000 + 6000*3 = 20,000
- Retrieve each DEPARTMENT, use index on EMPLOYEE: bD + (rD * (xSsn + 1)) = 10 + 50*5 = 260
- More efficient: use smaller file or file with high join selection factor.
• Index can be created specifically for join if none exists.

99. Sort-Merge Join Performance
𖦹 Efficient if both files are sorted by join attributes.
• Single pass through each file; block accesses = bE + bD = 2,010
- If unsorted, need to sort files: total cost ≈ bE + bD + bE log2 bE + bD log2 bD

100. General Partition-Hash Join
𖦹 Files R and S partitioned into M partitions using same hash function on join attributes.
• Partitioning phase:
- M buffers for partitions + 1 buffer for input file.
- Records hashed to memory buffer, appended to disk subfile if buffer full.
• Joining/probing phase:
- For each partition pair Ri, Si, use nested-loop join or in-memory hash table for smaller partition.
• Cost approximation: 3*(bR + bS) + bRES
- Uniform hash function required to avoid skew.

101. Hybrid Hash-Join
𖦹 Variation of partition-hash join; joining phase for one partition included in partitioning.
• Example OP6: smaller file DEPARTMENT.
- First partition resides entirely in memory; others on disk.
• Second file EMPLOYEE:
- Records hashing to first partition joined immediately.
- Records hashing to other partitions written to disk.
• Result: M-1 iterations needed in joining phase instead of M.
- Saves disk I/O by joining as many records as possible during partitioning phase.

102. PROJECT Operation Algorithms
𖦹 PROJECT π<attribute list>(R) selects only specified columns and removes duplicates.
• If <attribute list> includes a key, the result has same number of tuples as R.
• If <attribute list> does not include a key, duplicates must be removed.
- Duplicate elimination can be done by sorting and removing consecutive duplicates or using hashing in memory.
- SQL by default does not remove duplicates; DISTINCT keyword is required.

103. Set Operations Algorithms
𖦹 Set operations include UNION, INTERSECTION, SET DIFFERENCE (MINUS/EXCEPT), and CARTESIAN PRODUCT.
• CARTESIAN PRODUCT R × S produces n * m records with j + k attributes; expensive to compute.
- Prefer using join instead of CARTESIAN PRODUCT during query optimization.
• UNION, INTERSECTION, SET DIFFERENCE require type-compatible relations.
- Implemented using sort-merge: sort both relations and scan concurrently.
- For UNION: keep one copy if tuple exists in both relations.
- For INTERSECTION: keep only tuples present in both relations.
• Hashing can also implement these operations:
- For R ∪ S: hash R, probe S, avoid duplicates.
- For R ∩ S: hash R, probe S, keep only matching records.
- For R – S: hash R, probe S, remove matching records from R.

104. Anti-Join for SET DIFFERENCE
𖦹 MINUS or EXCEPT in SQL can be implemented using anti-join.
• Example:
- SQL: SELECT Dnumber FROM DEPARTMENT MINUS SELECT Dno FROM EMPLOYEE
- Anti-join conversion:
<pink>SELECT</pink> <yellow>DISTINCT DEPARTMENT.Dnumber</yellow>
<pink>FROM</pink> <yellow>DEPARTMENT, EMPLOYEE</yellow>
<pink>WHERE</pink> <yellow>DEPARTMENT.Dnumber A = EMPLOYEE.Dno</yellow>
- "A=" represents anti-join: left table DEPARTMENT, right table EMPLOYEE.

𖦹 SQL variations for sets and multisets:
• UNION, INTERSECTION, EXCEPT/MINUS: eliminate duplicates.
• UNION ALL, INTERSECTION ALL, EXCEPT ALL: retain duplicates (multisets).
- Example with INTERSECT ALL:
<pink>SELECT</pink> <yellow>Dno</yellow> <pink>FROM</pink> <yellow>EMPLOYEE</yellow>
<pink>INTERSECT ALL</pink>
<pink>SELECT</pink> <yellow>Dum</yellow> <pink>FROM</pink> <yellow>PROJECT</yellow>
- Semi-join can implement INTERSECT ALL while controlling duplicates:
<pink>SELECT</pink> <yellow>DISTINCT EMPLOYEE.Dno</yellow>
<pink>FROM</pink> <yellow>DEPARTMENT, EMPLOYEE</yellow>
<pink>WHERE</pink> <yellow>EMPLOYEE.Dno S = DEPARTMENT.Dnumber</yellow>

104. Implementing Aggregate Operations
𖦹 Aggregate operators (MIN, MAX, COUNT, AVERAGE, SUM) can be computed via table scan or using indexes.
• MAX(Salary) can be retrieved efficiently using a B+-tree index by following the rightmost pointer from root to leaf.
• MIN(Salary) uses the leftmost pointer in a B+-tree index.
• SUM and AVERAGE can use a dense index where each record has an index entry.
- For non-dense indexes, count of records per index value is needed.
• COUNT(*) can be retrieved from catalog without scanning the table.
• GROUP BY requires partitioning tuples by the grouping attribute using sorting or hashing.
- Aggregates are then applied to each group.
• If a clustering index exists on the grouping attribute, records are already grouped.

105. Implementing Different Types of JOINs
𖦹 Variations of JOIN include outer joins, semi-joins, and anti-joins.

𖦹 Outer Joins
• Types: left outer join, right outer join, full outer join.
• Example of left outer join:
<pink>SELECT</pink> <yellow>E.Lname, E.Fname, D.Dname</yellow>
<pink>FROM</pink> <yellow>EMPLOYEE E LEFT OUTER JOIN DEPARTMENT D ON E.Dno = D.Dnumber</yellow>
- Includes all tuples from left table; unmatched tuples have NULLs in right table fields.
• Can be computed by modifying join algorithms (nested-loop, single-loop, sort-merge, hash-join).
• Algebraic approach:
1. TEMP1 ← πLname, Fname, Dname (EMPLOYEE Dno=Dnumber DEPARTMENT)
2. TEMP2 ← πLname, Fname (EMPLOYEE) − πLname, Fname (TEMP1)
- Step 2 uses anti-join to find unmatched tuples.
3. TEMP2 ← TEMP2 × NULL
4. RESULT ← TEMP1 ∪ TEMP2
- Right outer join can be converted to left outer join by switching operands.
- Full outer join pads unmatched tuples from both relations.

𖦹 Semi-Join and Anti-Join
• Semi-join: selects outer tuple as soon as first match is found in inner relation.
• Anti-join: rejects outer tuple as soon as first match is found.
- Can be implemented as extensions of standard join algorithms.

𖦹 Non-Equi-Join (Theta-Join)
• Join condition uses inequality operators (<, >, ≥, ≤, ≠).
- All join methods except hash-based algorithms are applicable.

106. Combining Operations Using Pipelining
𖦹 Materialized evaluation executes one operation at a time, storing intermediate results on disk.
- This creates excessive overhead and delays query processing.
𖦹 Pipelining avoids storing temporary files by combining multiple operations into a single algorithm.
• Example: JOIN combined with two SELECT operations and a final PROJECT operation can produce one result file directly.
𖦹 Pipelined evaluation allows output tuples from one operation to be fed immediately to the next operation.
- Benefits:
  • Reduces cost and time of writing intermediate results to disk.
  • Allows early generation of result tuples while other operations continue processing.

107. Iterators for Implementing Physical Operations
𖦹 Iterators output one tuple at a time instead of generating full relations as temporary files.
• This contrasts with materialization where entire relations are stored and read back.
𖦹 Query plan execution invokes iterators in a specific order, allowing multiple iterators to be active simultaneously.
𖦹 Iterator interface methods:
1. <pink>Open()</pink>
- Initializes operator, allocates buffers, initializes data structures, and passes selection conditions.
2. <pink>Get_Next()</pink>
- Calls <pink>Get_Next()</pink> on input arguments, applies operation-specific code, returns next output tuple, updates iterator state.
3. <pink>Close()</pink>
- Ends iteration, calls <pink>Close()</pink> on input arguments.
𖦹 Iterators can be implemented as classes with these methods.
- Pipelining is effective if tuples can be processed completely as received.
- If multiple passes over input are required, materialization is needed, reducing pipelining benefit.
𖦹 Iterators can also be applied to access methods.
- Example: B+-tree or hash-based index can be implemented as an iterator producing tuples that meet selection conditions passed to <pink>Open()</pink>.

116. Query Trees and Heuristics for Query Optimization
𖦹 Heuristic rules are applied to the internal representation of a query to improve performance.
𖦹 SQL queries are first parsed into a data structure (query tree or query graph) and then optimized.
𖦹 Optimized query representation determines the query execution strategy.
𖦹 Query execution plans are generated based on access paths of the files involved.
𖦹 Main heuristic: Apply SELECT and PROJECT operations before JOIN or other binary operations to reduce file size and improve efficiency.

117. Notation for Query Trees and Query Graphs
𖦹 A query tree represents an extended relational algebra expression.
• Leaf nodes represent input relations of the query.
• Internal nodes represent relational algebra operations.
- Execution starts from leaf nodes and proceeds to root, producing the final result relation.
IMG_URL query_tree.png
𖦹 A query graph represents a relational calculus expression.
• Relation nodes are displayed as single circles.
• Constant nodes are displayed as double circles or ovals.
• Selection and join conditions are represented by edges.
• Attributes to retrieve are shown in square brackets.
𖦹 Query graph does not specify the order of operations; query trees indicate execution order.

118. Example: Query Q2
𖦹 Task: Retrieve project number, controlling department number, and department manager’s last name, address, and birthdate for projects located in ‘Stafford’.
𖦹 Relational algebra expression:
• πPnumber, Dnum, Lname, Address, Bdate (((σPlocation=‘Stafford’(<yellow>PROJECT</yellow>))
 Dnum=Dnumber(<yellow>DEPARTMENT</yellow>)) Mgr_ssn=Ssn(<yellow>EMPLOYEE</yellow>))
𖦹 SQL query:
• <pink>SELECT</pink> P.Pnumber, P.Dnum, E.Lname, E.Address, E.Bdate
  <pink>FROM</pink> <yellow>PROJECT</yellow> P, <yellow>DEPARTMENT</yellow> D, <yellow>EMPLOYEE</yellow> E
  <pink>WHERE</pink> P.Dnum=D.Dnumber AND D.Mgr_ssn=E.Ssn AND P.Plocation='Stafford';
𖦹 Execution order in query tree:
• Node (1) executes before node (2) because its resulting tuples are needed.
• Node (2) executes before node (3), and so on.

119. Advantages of Query Trees
𖦹 Query trees explicitly show the order of operations for execution.
𖦹 More practical than query graphs for optimizers since they need to indicate execution sequence.
𖦹 Query graphs are neutral and useful for representation but do not indicate operation order.

121. Example of Query Transformation
𖦹 Query: Find last names of employees born after 1957 who work on project ‘Aquarius’.
• SQL: <pink>SELECT</pink> E.Lname <pink>FROM</pink> <yellow>EMPLOYEE</yellow> E, <yellow>WORKS_ON</yellow> W, <yellow>PROJECT</yellow> P <pink>WHERE</pink> P.Pname='Aquarius' AND P.Pnumber=W.Pno AND E.Essn=W.Ssn AND E.Bdate>'1957-12-31';
𖦹 Initial query tree creates a large CARTESIAN PRODUCT of all relations.
𖦹 Step-by-step improvements:
• Apply SELECT operations early to reduce tuples.
• Reorder EMPLOYEE and PROJECT relations using key attribute info.
• Replace CARTESIAN PRODUCT followed by SELECT with JOIN operations.
• Apply PROJECT operations early to reduce columns in intermediate relations.

122. Transformation Rules for Relational Algebra
𖦹 Cascade of σ: Break conjunctive selection into sequence of σ operations.
𖦹 Commutativity of σ: σc1(σc2(R)) ≡ σc2(σc1(R))
𖦹 Cascade of π: Only last π matters; earlier ones can be ignored.
𖦹 Commuting σ with π: If σ involves attributes in projection, they can be swapped.
𖦹 Commutativity of join and ×: R <pink>JOIN</pink>c S ≡ S <pink>JOIN</pink>c R ; R × S ≡ S × R
𖦹 Commuting σ with join or ×: σc(R <pink>JOIN</pink>c S) ≡ (σc(R)) <pink>JOIN</pink>c S
𖦹 Commuting π with join or ×: πL(R <pink>JOIN</pink>c S) ≡ (πA(R)) <pink>JOIN</pink>c (πB(S)) with adjustments for join attributes.
𖦹 Commutativity of set operations: ∪ and ∩ are commutative; − is not.
𖦹 Associativity of <pink>JOIN</pink>, ×, ∪, ∩: (R θ S) θ T ≡ R θ (S θ T)
𖦹 Commuting σ with set operations: σc(R θ S) ≡ (σc(R)) θ (σc(S))
𖦹 π operation commutes with ∪: πL(R ∪ S) ≡ (πL(R)) ∪ (πL(S))
𖦹 Converting σ following × into JOIN: σc(R × S) ≡ R <pink>JOIN</pink>c S
𖦹 Pushing σ with − or ∩: Apply σc to only relevant relation(s).
𖦹 Trivial transformations: R ∪ ∅ = R ; σtrue(R) = R
𖦹 Boolean algebra rules: NOT(c1 AND c2) ≡ (NOT c1) OR (NOT c2); NOT(c1 OR c2) ≡ (NOT c1) AND (NOT c2)

123. Heuristic Algebraic Optimization Algorithm
𖦹 Step 1: Break conjunctive SELECT operations into a cascade of individual σ operations.
𖦹 Step 2: Move each SELECT operation down the tree using commutativity rules.
• If condition involves one table, move to leaf node of that table.
• If condition involves two tables (join), move after tables are combined.
𖦹 Step 3: Rearrange leaf nodes using commutativity and associativity rules.
• Execute most restrictive SELECT operations first (fewest tuples or smallest size).
• Avoid unnecessary CARTESIAN PRODUCT operations.
𖦹 Step 4: Combine CARTESIAN PRODUCT with SELECT into JOIN if condition represents a join.
𖦹 Step 5: Move PROJECT operations down the tree.
• Keep only attributes needed for subsequent operations.
𖦹 Step 6: Identify subtrees that can be executed by a single algorithm and group operations.

124. Summary of Heuristics
𖦹 Apply operations that reduce intermediate result size early: SELECT for tuples, PROJECT for attributes.
𖦹 Execute the most restrictive SELECT and JOIN operations first.
𖦹 Reorder leaf nodes carefully to avoid Cartesian products while maintaining efficiency.

125. Choice of Query Execution Plans
𖦹 Alternatives for Query Evaluation
• An execution plan contains access methods for each relation and algorithms for relational operators.
- Example: Query Q1: πFname, Lname, Address(σDname='Research'(DEPARTMENT) Dnumber=Dno EMPLOYEE)
- Optimizer choices: index search for SELECT, index-based nested-loop join, scan for PROJECT.
• Materialized vs Pipelined evaluation
- Materialized: store operation result as temporary relation, then read for next operation.
- Pipelined: forward tuples directly to next operation, reducing disk I/O.
• Query tree transformations
- Heuristic-based or cost-based methods optimize execution time and resources.
• Nested subqueries in WHERE or FROM
- Correlated subquery: inner references outer relation.
- FROM subquery: acts like a view.
- Optimizer transforms and rewrites nested subqueries.

126. Nested Subquery Optimization
𖦹 Evaluation strategy
• Outer query uses value from inner query (e.g., MAX(Salary)).
- Efficient methods: use index or catalog if available; otherwise linear search.
• Correlated subqueries
- Inner query depends on outer query variables.
- Example: EXISTS (SELECT * FROM DEPARTMENT D WHERE D.Dnumber = E.Dno AND D.Zipcode=30332)
- Naïve evaluation: inner query executed per outer tuple (inefficient)
- Optimizer converts nested subquery to join (unnesting).
• Semi-join
- Used when inner join would produce extra tuples.
- Example: IN operator converted using semi-join.
- Anti-join: used for NOT IN connectors.
- Unnesting/decorrelation improves performance for complex nested queries.

127. Subquery (View) Merging Transformation
𖦹 Inline view or predefined view in FROM clause
• Optimizer can merge view with outer query to enable more join orders and index usage.
- Example: EMP, DEPT, BLDG joined with inline view V can be merged into single query joining all three tables.
• GROUP-BY view merging
- Decision to merge depends on optimizer cost estimation.
- Early vs delayed Group By: early reduces data for joins, delayed may reduce grouping cost.
- Example: SALES, CUST, PRODUCT with CP_BOUGHT_VIEW materialized or merged for efficient query execution.

128. Materialized Views
𖦹 Concept
• Stores result of a query to avoid recomputation.
• Useful for expensive operations like join and aggregation.
• Can be temporary or permanent.
𖦹 Update strategies
• Immediate: update view on any base relation change.
• Lazy: recompute only on demand.
• Periodic: update at regular intervals.
• Triggers may automate updates.
𖦹 Incremental View Maintenance
• Update view only with net changes from inserted/deleted tuples.
• Join: vnew = (r ∪ ri) ⨝ s or adjust for deletions.
• Selection: vnew = vold ∪ σC ri or vnew = vold − σC rd.
• Projection: maintain counts for tuples; update on insert/delete.
• Intersection: insert if in other relation; delete if present.
• Aggregation (Group By)
- Count: increment/decrement per group.
- Sum: maintain sum and count per group; adjust on insert/delete.
- Average: maintain sum and count to compute average.
- Max/Min: track max value and count; recompute if max deleted.
𖦹 Use in queries
• Replace query components with materialized view to save computation.
• Example: using view for R ⨝ S or πLR to avoid recomputation; indexes may further optimize access.
