1. File organization and basic data access
𖦹 A collection of records is stored in files, and each file consists of pages.
𖦹 The file layer arranges data to support fast access to needed records.
𖦹 A file organization is a way of arranging records on disk.
• Each organization makes some operations efficient and others slower.
𖦹 Example: Sorting employee records by age makes age-based retrieval fast.
• But maintaining sorted order is expensive when records change.
𖦹 If we want to retrieve employees with salary > 5000, we must scan the whole file.
𖦹 Indexing helps support multiple query types efficiently.
𖦹 Choosing the right file organization affects performance.
𖦹 Choosing good indexes is one of the most effective performance improvements.

2. Data on external storage
𖦹 Database data must persist across program executions, so it is stored on external storage.
𖦹 Data is stored on disks or tapes and brought into memory when needed.
𖦹 A page is the unit of data transfer between disk and memory, often 4KB or 8KB.
𖦹 Page I/O cost dominates the total cost of operations, so DBMS optimizes to reduce it.
𖦹 Disks allow random page access at a roughly fixed cost.
• Reading pages sequentially is faster than reading them in random order.
𖦹 Tapes allow only sequential access and are mainly used for archival storage.
𖦹 Every record has a unique record id (rid).
• Using rid, the system can locate the disk page containing the record.

3. Buffer manager and disk space manager
𖦹 The buffer manager handles moving pages between disk and memory.
• When a page is needed, the file layer requests it from the buffer manager.
𖦹 The disk space manager allocates and tracks disk pages.
• When new space is needed, it allocates pages.
• When pages are no longer needed, it marks them free and reuses them later.
𖦹 The files and access methods layer relies on these managers for storage and retrieval.

4. File organizations and indexing
𖦹 A file of records is handled by the files and access methods layer.
𖦹 The file layer can create, delete, insert records, delete records, and scan through all records.
• A relation is stored as a file of records.
𖦹 Records are stored across disk pages, and the file layer tracks allocated pages and available space.
𖦹 The simplest file organization is a heap file where records are stored in random order.
𖦹 An index is a data structure that supports fast retrieval based on search key fields.
• Additional indexes can be created on different search keys to support different search operations.

5. Data entries in an index
𖦹 A data entry (k*) in an index contains enough information to locate records with search key value k.
𖦹 Searching the index finds data entries, which are then used to locate data records.
𖦹 Three alternatives for storing data entries:
• Alternative (1): The data entry is the actual data record.
• Alternative (2): The data entry is a (k, rid) pair, where rid is the record id.
• Alternative (3): The data entry is a (k, rid-list) pair, where rid-list contains all matching record ids.
𖦹 Alternative (1) means the index itself is a file of data records.
𖦹 Alternatives (2) and (3) store pointers and are independent of how the main data file is organized.
𖦹 Alternative (3) uses space efficiently but has variable-length entries.
𖦹 Only one index should use Alternative (1) to avoid storing records multiple times.

6. Clustered indexes
𖦹 An index is clustered if the order of data records matches the index order.
𖦹 If the index uses Alternative (1), it is clustered by definition.
𖦹 Alternatives (2) and (3) are clustered only if the data file itself is sorted by the search key.
𖦹 In practice, files are rarely kept sorted, so clustered indexes usually use Alternative (1).
𖦹 A clustered index returns data pages that are close together for range queries, reducing page I/O.
𖦹 An unclustered index may require many separate page fetches for range queries.

7. Primary and secondary indexes
𖦹 An index on fields that include the primary key is called a primary index.
𖦹 Indexes on other fields are secondary indexes.
𖦹 Terminology varies; some define primary index as Alternative (1) and secondary as Alternatives (2) or (3).
𖦹 A primary index does not have duplicate search key values.
𖦹 Secondary indexes may contain duplicates.
𖦹 If no duplicates exist, the index is called a unique index.

8. Index data structures
𖦹 Data entries can be organized using hashing or tree-based structures.
𖦹 Either hashing or tree indexing can be used with any of the three data entry alternatives.

9. Hash-based indexing
𖦹 Hashing groups records into buckets based on a hash function applied to the search key.
𖦹 Each bucket consists of a primary page and possibly overflow pages linked in a chain.
𖦹 Searching:
• Apply the hash function to identify the correct bucket.
• Retrieve all pages in that bucket to find matching records.
𖦹 Insertion:
• Insert the record into its bucket.
• Overflow pages are added if the bucket is full.
𖦹 If we do not know the search key value, we must scan the entire file.
𖦹 Hash-based indexes usually allow retrieval of a page in one to two disk I/Os.
𖦹 Example:
• File of employees hashed on age uses Alternative (1), storing actual data records.
• Auxiliary index on sal uses Alternative (2), storing (sal, rid) pairs.

10. Tree-based indexing
𖦹 Data entries are stored in sorted order by search key.
𖦹 A hierarchical tree structure guides searches to the correct leaf page.
𖦹 Leaf level contains the data entries.
𖦹 Non-leaf nodes contain search key values and pointers to child nodes.
• Pointer to the left of key k leads to entries less than k.
• Pointer to the right of key k leads to entries greater than or equal to k.
𖦹 Leaf pages are linked to allow efficient scanning across a range of values.
𖦹 The search cost equals the height of the tree plus the number of leaf pages that satisfy the condition.
𖦹 B+ tree maintains equal path length from root to every leaf, ensuring balance.
𖦹 Tree height is usually small (often 3 or 4), so retrieval typically takes only a few disk I/Os.
𖦹 Fan-out refers to the average number of children of a non-leaf node.
• If fan-out is F and height is h, number of leaf pages is approximately F^h.
𖦹 With large fan-out (e.g., F ≥ 100), a tree of height 4 can index around 100 million leaf pages efficiently.

11. Fan-out and tree height
𖦹 The average number of children for a non-leaf node is called the fan-out.
𖦹 If every non-leaf node has n children, a tree of height h has n^h leaf pages.
𖦹 In practice, nodes have varying numbers of children, but the average F can approximate leaf pages as F^h.
𖦹 Typically, F is at least 100, so a tree of height 4 can have 100 million leaf pages.
𖦹 Searching such a tree requires 4 I/Os.
• This is much faster than a binary search, which would require over 25 I/Os for the same number of pages.

12. Comparison of File Organizations
𖦹 Basic file organizations considered:
• Heap file (randomly ordered records)
• Sorted file (records sorted on age, sal)
• Clustered B+ tree file (search key age, sal)
• Heap file with unclustered B+ tree index (age, sal)
• Heap file with unclustered hash index (age, sal)
- Choice of file organization affects operation costs. Sorting or indexing can improve performance for some operations.

13. Operations on Files
𖦹 Scan: Fetch all records from disk into memory.
𖦹 Search with Equality Selection: Fetch records matching exact values (e.g., age=23, sal=50).
𖦹 Search with Range Selection: Fetch records in a range (e.g., age>35).
𖦹 Insert: Find appropriate page, add record, write back, may affect other pages.
𖦹 Delete: Locate page and record, remove record, write back, may affect other pages.

14. Cost Model
𖦹 B = number of data pages, R = records per page.
𖦹 D = average time to read/write a page, C = average time to process a record.
𖦹 H = time to apply hash function, F = fan-out for tree indexes.
𖦹 I/O usually dominates cost; CPU is less significant.
𖦹 Simplified model counts pages read/written; ignores blocked access and other factors.

15. Heap Files
𖦹 Scan cost: B(D + RC)
𖦹 Equality selection (candidate key): 0.5B(D + RC)
𖦹 Equality selection (non-key): B(D + RC)
𖦹 Range selection: B(D + RC)
𖦹 Insert at end: 2D + C
𖦹 Delete by rid: D + C + D
- Deletion by condition may require scanning and modifying multiple pages.

16. Sorted Files
𖦹 Scan cost: B(D + RC)
𖦹 Equality selection matching sort order: Dlog2B + Clog2R
• Multiple matching records are contiguous; sequential retrieval is efficient.
𖦹 Range selection: Search first record (Dlog2B + Clog2R), then sequentially fetch pages.
𖦹 Insert: Search + read/write subsequent pages; cost ~ search + B(D + RC)
𖦹 Delete: Search + read/write subsequent pages to compact; same cost as insert.
- Sorting helps locate records efficiently but increases insertion/deletion cost.

17. Clustered Files
𖦹 Average page occupancy: 67%, total pages ~1.5B
𖦹 Scan: 1.5B(D + RC)
𖦹 Equality selection: DlogF1.5B + Clog2R
𖦹 Range selection: First record located as equality, then sequentially fetch leaf pages.
𖦹 Insert: Search + write modified leaf page; occasional splits ignored
𖦹 Delete: Search + modify/write pages; similar to insert
- Clustered files combine sorted order benefits with efficient insert/delete.

18. Heap File with Unclustered Tree Index
𖦹 Each data entry ~0.1 size of a record; leaf pages ~0.15B
𖦹 Scan: Leaf level cost 0.15B(D + 6.7RC) + fetch each record BR(D + C)
𖦹 Equality search: DlogF0.15B + Clog2(6.7R) + D
𖦹 Range search: Cost depends on number of qualifying records; each requires one I/O
𖦹 Insert: 2D + C (heap) + DlogF0.15B + Clog2(6.7R) + D (index)
𖦹 Delete: Locate record + index, then write modified pages; cost 2D + DlogF0.15B + Clog2(6.7R)
- Unclustered index fast for single-record search but expensive for scans/range queries.

19. Heap File with Unclustered Hash Index
𖦹 Each data entry ~0.1 size of record; pages ~0.125B; 80% occupancy
𖦹 Scan: Retrieve entries 0.125B(D + 8RC) + fetch each record BR(D + C), very expensive
𖦹 Equality search: H + 2D + 4RC
𖦹 Range search: Entire heap file scanned; cost B(D + RC)
𖦹 Insert: 2D + C (heap) + H + 2D + C (index)
𖦹 Delete: Locate record + index: H + 2D + 4RC, then write pages 2D
- Hash indexes are very fast for equality, do not support efficient range searches.

20. Comparison of I/O Costs
𖦹 Heap file: efficient storage, fast scan/insert, slow search/delete
𖦹 Sorted file: efficient storage, fast search, slow insert/delete
𖦹 Clustered file: fast search, efficient insert/delete, extra space overhead
𖦹 Unclustered tree/hash: fast search/insert/delete, slow scan/range search
𖦹 Hash slightly faster for equality, cannot do range search
- No single file organization is best for all operations.

21. INDEXES AND PERFORMANCE TUNING
𖦹 Indexes improve database performance and must be chosen based on expected workload.
𖦹 Index choice impacts performance and requires understanding query evaluation and concurrency control.
𖦹 Examples with multiple tables require knowledge of join algorithms and query plans.

22. Impact of the Workload
𖦹 Consider expected workload and common operations when choosing indexes.
𖦹 Indexes support efficient retrieval for selection conditions: equality and range.
• Hash-based indexes are good for equality selections but poor for range selections.
• Tree-based indexes support both equality and range selections efficiently.
𖦹 Both tree and hash indexes handle inserts, deletes, and updates efficiently.
𖦹 Tree-based indexes are better than fully sorted files for:
• Efficient inserts and deletes.
• Faster leaf page search than binary search in sorted files.
𖦹 Sorted files are faster for sequential page retrieval but expensive for inserts and deletes.
𖦹 ISAM combines sequential leaf allocation with fast search; inserts/deletes are better than sorted files but worse than B+ trees.

23. Clustered Index Organization
𖦹 Clustered index organizes the underlying data records; only one clustered index per table.
𖦹 Multiple unclustered indexes can exist on a data file.
𖦹 Clustered indexes are expensive to maintain when inserting into full leaf pages due to record movement and updates in other indexes.
𖦹 Clustering should be used sparingly for queries that benefit from sequential access.
𖦹 Hash indexes are unsuitable for clustered files because range queries cannot use them.
𖦹 Index-only evaluation can compute queries using just index entries without fetching data records.
𖦹 Example: Using a B+ tree index on age for SELECT <pink>FROM</pink> <yellow>Employees E</yellow> <pink>WHERE</pink> <yellow>E.age > 40</yellow>.
• Benefit depends on selectivity of the condition and whether the index is clustered.
𖦹 Queries on non-key fields may benefit from clustering if many records share the same value.
𖦹 Aggregate queries can use indexes (clustered or unclustered) without accessing full tuples.

24. Composite Search Keys
𖦹 A composite search key contains multiple fields, e.g., (age, sal) or (sal, age).
𖦹 Equality queries bind all fields in the search key to constants; hash indexes support only equality queries.
𖦹 Range queries bind only some fields; tree indexes support range queries on a prefix of the composite key.
𖦹 Composite keys allow broader query matching and increase opportunities for index-only evaluation.
𖦹 Drawbacks: updates affect any field in the key; composite indexes are larger and may increase tree levels.
𖦹 Design examples:
• SELECT <pink>FROM</pink> <yellow>Employees E</yellow> <pink>WHERE</pink> <yellow>E.age BETWEEN 20 AND 30 AND E.sal BETWEEN 3000 AND 5000</yellow>
- A composite B+ tree index on (age, sal) is effective; hash index cannot help.
• SELECT <pink>FROM</pink> <yellow>Employees E</yellow> <pink>WHERE</pink> <yellow>E.age = 25 AND E.sal BETWEEN 3000 AND 5000</yellow>
- Index on (age, sal) performs better than (sal, age) due to ordering of attributes.
• Aggregate queries benefit from composite indexes for index-only scans.

25. Index Specification in SQL: 1999
𖦹 SQL:1999 does not require support for indexes or provide standard commands to create/drop them.
𖦹 Commercial DBMS support indexes; example for creating a B+ tree index:
<pink>CREATE INDEX</pink> <yellow>IndAgeRating ON Students</yellow>
<pink>WITH STRUCTURE =</pink> <yellow>BTREE, KEY = (age, gpa)</yellow>
𖦹 The index automatically updates with inserts and deletes on the Students table.
